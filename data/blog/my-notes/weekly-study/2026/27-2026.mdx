### [追逐幻影跳跃：我们如何在低端安卓设备上让 Pingo 动画流畅运行 — Margelo 博客](https://blog.margelo.com/profiling-skia-reanimated-low-end-android)

本文详细分析了在廉价 120Hz 安卓设备上，一个看似正确的弹簧动画出现视觉卡顿的根本原因。问题并非动画逻辑本身，而是由渲染管线中的多个成本叠加导致帧交付延迟。文章按影响大小依次介绍了三个主要成本：Skia 在安卓上的合成方式（TextureView vs SurfaceView）、一个自循环的重新绘制错误，以及 React 状态管理导致的额外渲染。最终通过将画布从 TextureView 切换到 SurfaceView，并修复其他问题，彻底解决了卡顿。

- 🧩 **核心问题**：动画数学上完美（位置单调），但因帧交付时间不均匀（部分帧耗时超过 8.3ms），导致人眼感知到卡顿。问题在于“帧节奏”，而非“位置曲线”。
- 🖼️ **最大成本：Skia 合成方式**：默认的`TextureView`导致每帧 GPU 需对画布像素进行 2-3 次处理（渲染、上传、合成），在廉价 120Hz 设备上远超 8.3ms 预算。`dumpsys gfxinfo`显示 85% 的帧卡顿，其中 79% 归因于“慢位图上传”。
- 🔬 **诊断方法**：将屏幕强制降至 60Hz 后，卡顿率从 85% 骤降至 15%，证实了每帧 GPU 成本是固定的，且超出了 120Hz 的预算。
- ⚙️ **关键修复**：将`<Canvas>`的`opaque`属性设为`true`（安卓上），使其从`SkiaTextureView`切换为`SkiaSurfaceView`。这为画布创建了独立的`SurfaceFlinger`层，消除了每帧的纹理上传和 GPU 合成，让硬件合成器直接处理，RenderThread CPU 使用率下降 50-65%。
- 🔄 **第二成本：自循环重绘**：`usePathValue`和`usePathInterpolation`钩子会创建一个自循环，每次重绘都重新脏化自身，导致画布在无交互时也持续重绘。通过改用缓冲区变异（`useDerivedValue`返回稳定引用）修复，空闲帧时间从 17-32ms 降至约 9ms。
- ⚡ **第三成本：React 状态搅动**：`useEffect`依赖了每次轮询都创建新对象的 Zustand store 值，导致`React.memo`失效，触发不必要的画布重绘。通过基于内容而非对象引用来记忆化修复。
- 🤖 **自动化二分法**：使用 LLM 代理（Claude Code）并行构建、运行和测量数百个变体，快速定位次要成本。人类负责提出假设和判断结果，机器负责执行重复循环。
- 📊 **最终效果**：在廉价 120Hz 设备上，动画从 85% 卡顿帧变为流畅，用户不再感知到跳跃。所有修复中，`SurfaceView`切换贡献了绝大部分性能提升。

### [React 测试中常让工程师困惑的问题 | 如何测试前端](https://howtotestfrontend.com/resources/react-testing-interview-questions)

本文探讨了前端测试中常见的陷阱与最佳实践，尤其针对 React 应用和 React Testing Library。文章通过面试问题形式，剖析了“永不失败的测试”、过度模拟、测试实现细节等关键问题，并提供了具体解决方案，如使用假定时器、正确模拟 Web API 等。

- 🧪 **永不失败的测试**：测试硬编码文本或错误断言（如 `expect(container).toBeTruthy()`）毫无价值，应避免。若测试在实现被删除后仍通过，则说明测试无效。
- 🔄 **测试实现细节**：如果重构内部代码（如从 `useState` 改为 `useReducer`）导致测试失败，则说明测试了实现细节。应只通过用户交互（如点击按钮后文本变化）来验证。
- ⏱️ **测试防抖输入**：使用假定时器（`jest.useFakeTimers()`）和配置 `userEvent.setup({ advanceTimers })`，并在 `act()` 中推进时间，而非依赖真实时间。
- 🎭 **过度模拟的危害**：模拟过多会导致测试与真实代码脱节，如模拟 API 返回数据后只验证模拟本身。应仅模拟外部依赖（如浏览器 API、第三方模块）。
- 🌐 **测试 IntersectionObserver**：`jsdom` 不支持该 API，需手动模拟或使用真实浏览器（如 Vitest Browser Mode 或 E2E 测试）。提供完整模拟代码示例。
- 🖱️ **测试拖拽功能**：优先使用键盘事件（`fireEvent.keyDown`）模拟拖拽，因为`jsdom`无真实布局。鼠标拖拽应留给 E2E 测试。
- 🚫 **E2E 测试的过度使用**：E2E 测试慢、易不稳定，应只覆盖关键流程。组件细节和边缘用例更适合单元/集成测试。
- 📚 **RTL 基础问题**：熟悉 `getBy` vs `queryBy` vs `findBy`、`userEvent` vs `fireEvent`、`data-testid` 作为最后手段，以及 `waitFor` 的用法。
- ⚠️ **`act()` 的作用**：确保所有状态更新和重渲染完成。忽略 `not wrapped in act()` 警告可能导致假阳性。`userEvent` 和 `waitFor` 内部已包含 `act()`。

### [gnhf](https://github.com/kunchenguid/gnhf)

gnhf 是一个开源自动化研究编排工具，可在您睡眠期间让 AI 代理持续工作，每次迭代都会向目标推进一个小而确定的提交，醒来时即可获得干净的分支和完整的工作日志。

- 💤 **自动化夜间工作**：睡前只需一条命令，代理就会持续运行直到你停止或达到配置上限，醒来即可看到干净的代码分支和完整日志
- 🔄 **智能迭代机制**：每次成功迭代自动提交，失败回滚（提交失败保留未提交代码供修复），代理错误指数退避重试，连续失败 3 次自动中止
- 📊 **实时状态与总结**：终端标题实时显示状态、令牌数和提交数；运行结束时输出包含耗时、分支、迭代数、令牌数、差异统计和审查命令的永久摘要
- 🔧 **多代理支持**：开箱支持 Claude Code、Codex、Rovo Dev、OpenCode、GitHub Copilot CLI、Pi 和 ACP 目标，可通过配置文件切换
- 🌳 **并发工作树模式**：通过 `--worktree` 在隔离的 git 工作树中同时运行多个代理，互不干扰，适合同时实现多个功能
- 🔄 **断点续跑**：支持从现有 gnhf/ 分支恢复运行，提示不同时可选择更新提示继续、新建分支或退出
- 🚀 **实时分支模式**：`--current-branch --push` 组合可在当前分支直接提交并推送，适合需要实时部署或监看的项目
- 🛡️ **优雅中断与睡眠保护**：首次 Ctrl+C 优雅停止（等当前迭代完成），第二次立即强制停止；支持 macOS/Linux/Windows 系统睡眠保护
- 📝 **共享记忆与元数据**：代理通过 notes.md 文件跨迭代通信，运行元数据存储在 .gnhf/runs/ 目录下，不污染工作分支
- 🔧 **高度可配置**：通过 `~/.gnhf/config.yml` 设置默认代理、自定义路径、代理参数、提交约定等；CLI 标志可覆盖配置文件

### [扩展工程：所有权优先于招聘 | 伊多·格林](https://greenido.dev/2026/06/11/what-changes-at-20-50-and-200-engineers/)

扩展工程团队的关键在于扩展所有权，而非单纯增加人手。随着团队规模从 20 人增长到 200 人，协调成本会随之增加，而清晰的所有权是保持效率的核心。

- 🚀 **前 20 人：优化构建者** – 团队应拥有端到端的服务所有权，包括独立负责部署、值班和故障修复，避免过早设立平台团队。
- ⚠️ **50 人左右：协调成本出现** – 沟通不再自然发生，依赖关系变得复杂。需警惕“组织表演性”流程，只引入解决真实问题的流程。
- 🔍 **代码审查成为新瓶颈** – AI 时代，写代码变便宜，但审查、维护和保持架构完整性变难。重点应从“写更多代码”转向“保持对代码的信心”。
- 🏗️ **200 人左右：扩展所有权是核心工作** – 管理重点从软件开发转向“团队系统”设计。组织设计的重要性不亚于技术架构。
- 🌐 **协调的真实成本** – 跨团队、跨时区的依赖、共享责任和不清晰的所有权边界会带来“税收”。目标是持续最小化组织阻力，同时保持对齐。
- 🛠️ **平台团队的正确时机** – 在 80-100 人之后，当重复劳动和可靠性需求足够高时，平台团队才能创造真正的杠杆效应，且应像产品团队一样服务内部客户。
- 📞 **值班揭示真相** – 健康组织将值班视为学习机制，通过反馈循环改进系统；不健康组织则将其视为惩罚。
- 🏢 **上市公司的转变** – 需平衡速度与可预测性。避免过度流程化，在保持质量的同时维护所有权文化。
- 🤖 **AI 改变与不变** – AI 加速了代码生成，但并未改变所有权的重要性，反而因代码量激增而需要更强的所有权模型。
- 💡 **核心思想** – 扩展工程不是增加工程师，而是确保每位工程师清楚自己拥有什么、为何拥有以及当故障发生时会发生什么。清晰的目标、优先级和所有权是成功的关键。

### [维持共享现实：过往技术浪潮如何影响战略 | 作者：惠特尼·齐默尔曼 | 2026 年 5 月 | Medium](https://whitneyzim.medium.com/sustaining-a-shared-reality-how-past-technology-waves-have-impacted-strategy-c5965139341f)

历史经验表明，每一次信息技术浪潮都会让战略分析变得更容易，但也容易导致分析脱离决策，使企业陷入“分析瘫痪”。生成式 AI 是又一次强大变革，理解过往教训能帮助战略领导者更好地利用它。

- 📉 **分析脱离决策的陷阱**：1972 年，麦肯锡合伙人 Gerstner 指出，数据处理的普及让战略规划成为“浪费时间的玩物”，因为分析未能与真实决策结合，CEO 必须亲自参与。
- 🏢 **IBM 的教训**：1993 年 Gerstner 接手 IBM 时，公司拥有 339 项客户满意度调查，却因销售团队自选样本而系统性地误导战略。他通过“熊抱行动”强制高管亲自拜访客户，重建了共享认知。
- 🧠 **自动化悖论**：自动化接管常规任务后，人类会疏于练习关键技能。研究表明，“自动化自满”是注意力问题，无法通过意志克服，导致领导者在战略框架、辩论和选择上脱节。
- 🤝 **共享心智模型的价值**：团队通过协作分析（而非接收成品报告）建立的共享认知，是应对不确定性的关键。Gerstner 的“熊抱行动”实质是强制高管共同构建对现实的理解。
- ⏳ **节奏错配**：分析层（快）与决策层（慢）的失衡是危机根源。AI 加速分析，但战略的框架构建、辩论和承诺仍需人类节奏，过快产出会压垮团队的吸收能力。
- 🛠️ **战略领导者的行动建议**：保持深度参与（如 NVIDIA 的实时白板辩论）；不急于自动化，先培养团队质疑能力；投资于问题分解而非分析数量；警惕机器般完美的输出，珍视思考的“不完美痕迹”。
- 🎨 **艺术隐喻**：拉斯金在《威尼斯之石》中赞美哥特柱头的“不完美”，认为每个凿痕都记录着工匠的思考。战略亦如此——最宝贵的部分正是人类判断、诠释和选择的痕迹。
