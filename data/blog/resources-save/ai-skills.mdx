## Skill 编写

[add-skill](https://github.com/vercel-labs/add-skill)

一个用于为编码智能体（如 OpenCode、Claude Code、Codex 和 Cursor）安装技能的命令行工具。技能通过包含 YAML 前言的 SKILL.md 文件定义，可扩展智能体的功能，例如生成发布说明或集成外部工具。工具支持从 GitHub、GitLab 等源码安装，并提供全局或项目级安装选项。

[Skill_Seekers - 制作 skill 的工具](https://github.com/yusufkaraaslan/Skill_Seekers)

[claude-skills - 生成 skill 的 skill - 元 skill](https://github.com/tukuaiai/vibe-coding-cn/blob/main/i18n/zh/skills/claude-skills/SKILL.md)

[skill-prompt-generator - 基于 Claude Code Skills 的智能提示词生成系统](https://github.com/huangserva/skill-prompt-generator)

[colleague-skill - 把同事蒸馏成 AI Skill](https://github.com/titanwings/colleague-skill)

一个偏“人格 + 工作方式”建模的 Skill 生成器。它支持基于飞书消息、钉钉文档、邮件、截图和主观描述，生成可在 Claude Code / OpenClaw 中调用的“同事 Skill”，让智能体尽量复现某位同事的技术规范、表达风格和决策习惯。

- 不只是生成提示词：产物同时包含 Work Skill 和 Persona 两部分，强调“先按角色判断，再按能力执行”。
- 适合沉淀团队经验：能把某位同事的 code review 风格、接口规范、沟通方式和职场行为抽成可复用资产。
- 支持持续演化：可以增量追加材料、对话纠正设定，并保留版本以便回滚。

## Skill 管理工具

[skills-npm](https://github.com/antfu/skills-npm)
skills-npm 是一个命令行工具，用于发现 npm 包中内置的智能体技能，并通过创建符号链接供编程智能体使用。该项目旨在解决现有技能分发方式中的摩擦，如仅支持 Git 源、版本不匹配和手动管理问题，提出将技能直接打包在 npm 包中的新约定。

[skillman](https://github.com/pi0/skillman)

[rulesync](https://github.com/dyoshikawa/rulesync)
Rulesync 是一个 Node.js 命令行工具，旨在通过统一的 AI 规则文件，自动为多种主流 AI 开发工具生成配置文件。它支持选择性生成、全面的导入/导出功能，并能管理规则、命令、MCP 配置、忽略文件、子代理和技能，从而实现跨工具开发规则的一致性

[dotagents](https://github.com/iannuttall/dotagents)

[Agent-Reach - 给 Agent 装上全网搜索与阅读能力](https://github.com/Panniantong/Agent-Reach)

一个面向 Claude Code、OpenClaw、Cursor、Codex 等编码代理的联网 Skill + 安装器，目标不是自己包一层搜索服务，而是把 Twitter/X、Reddit、YouTube、GitHub、B 站、小红书、抖音、微信公众号、RSS 等渠道统一接进 agent 工作流里。

- 更像“联网能力总线”：安装后 agent 会直接调用各平台对应的上游工具，而不是被一个黑盒 wrapper 包住。
- 覆盖面很广：除了通用网页和 GitHub，还兼顾中文平台与内容源，适合做资料收集、竞品检索和多平台内容阅读。
- 对 agent 场景友好：既提供 CLI，也提供可安装的 Skill，适合在 Claude Code / OpenClaw 这类环境里按触发词自动调用。
- 成本门槛低：项目主打零 API 费用，更多是处理工具选择、安装和环境诊断，减少“明明能搜却接不起来”的配置摩擦。

[Waza - 面向完整工程师的 Claude Code Skills](https://github.com/tw93/Waza)

一个偏“少而精”的 Claude Code Skill 集合，围绕设计、排障、检查、写作、学习和资料阅读这些工程习惯设计，不追求大而全，而是把高频动作做成触发即用的稳定工作流。

- 技能集合克制：核心围绕 `/think`、`/design`、`/check`、`/hunt`、`/write`、`/learn`、`/read`、`/health` 等入口展开，适合不想维护庞大规则体系的人。
- 强调工程习惯而非功能堆叠：把需求挑战、系统化调试、交付前自检、英文表达和资料消化都沉淀成可复用动作。
- 安装门槛低：支持通过 `npx skills add tw93/Waza -g -y` 一键全量安装，也支持按单个 skill 单独安装。
- 适合做个人基线：如果已经在用 Claude Code，但还没有一套轻量、清晰、可长期维护的 skill 组合，Waza 很适合作为起点。

# UI Skills

Opinionated constraints for building better interfaces with agents.

## Stack

- MUST use Tailwind CSS defaults (spacing, radius, shadows) before custom values
- MUST use `motion/react` (formerly `framer-motion`) when JavaScript animation is required
- SHOULD use `tw-animate-css` for entrance and micro-animations in Tailwind CSS
- MUST use `cn` utility (`clsx` + `tailwind-merge`) for class logic

## Components

- MUST use accessible component primitives for anything with keyboard or focus behavior (`Base UI`, `React Aria`, `Radix`)
- MUST use the project’s existing component primitives first
- NEVER mix primitive systems within the same interaction surface
- SHOULD prefer [`Base UI`](https://base-ui.com/react/components) for new primitives if compatible with the stack
- MUST add an `aria-label` to icon-only buttons
- NEVER rebuild keyboard or focus behavior by hand unless explicitly requested

## Interaction

- MUST use an `AlertDialog` for destructive or irreversible actions
- SHOULD use structural skeletons for loading states
- NEVER use `h-screen`, use `h-dvh`
- MUST respect `safe-area-inset` for fixed elements
- MUST show errors next to where the action happens
- NEVER block paste in `input` or `textarea` elements

## Animation

- NEVER add animation unless it is explicitly requested
- MUST animate only compositor props (`transform`, `opacity`)
- NEVER animate layout properties (`width`, `height`, `top`, `left`, `margin`, `padding`)
- SHOULD avoid animating paint properties (`background`, `color`) except for small, local UI (text, icons)
- SHOULD use `ease-out` on entrance
- NEVER exceed `200ms` for interaction feedback
- MUST pause looping animations when off-screen
- MUST respect `prefers-reduced-motion`
- NEVER introduce custom easing curves unless explicitly requested
- SHOULD avoid animating large images or full-screen surfaces

## Typography

- MUST use `text-balance` for headings and `text-pretty` for body/paragraphs
- MUST use `tabular-nums` for data
- SHOULD use `truncate` or `line-clamp` for dense UI
- NEVER modify `letter-spacing` (`tracking-`) unless explicitly requested

## Layout

- MUST use a fixed `z-index` scale (no arbitrary `z-x`)
- SHOULD use `size-x` for square elements instead of `w-x` + `h-x`

## Performance

- NEVER animate large `blur()` or `backdrop-filter` surfaces
- NEVER apply `will-change` outside an active animation
- NEVER use `useEffect` for anything that can be expressed as render logic

## Design

- NEVER use gradients unless explicitly requested
- NEVER use purple or multicolor gradients
- NEVER use glow effects as primary affordances
- SHOULD use Tailwind CSS default shadow scale unless explicitly requested
- MUST give empty states one clear next action
- SHOULD limit accent color usage to one per view
- SHOULD use existing theme or Tailwind CSS color tokens before introducing new ones
